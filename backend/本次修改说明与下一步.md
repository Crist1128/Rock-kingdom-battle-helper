# 本次修改说明与下一步开发计划

## 一、本次修改目标

本次修改把项目从“后端基础骨架”推进到“手动输入 MVP 后端闭环”。重点是先把战斗数据流跑通：己方配置、双方阵容、敌方候选、统一状态、伤害事件、快照和公式占位推算。

本次没有实现真实伤害公式，也没有接入图像识别。所有与计算有关但公式未确认的地方，都以稳定接口和 `formula_unavailable` 占位结果保留，避免用错误公式污染后续候选推算。

## 二、主要新增内容

### 1. 己方配置管理

新增文件：

- `app/schemas/player_build.py`
- `app/services/player_build_service.py`
- `app/api/v1/endpoints/player_builds.py`

新增接口：

```http
POST /api/v1/player-builds
GET  /api/v1/player-builds
GET  /api/v1/player-builds/{build_id}
```

功能说明：

- 创建己方精灵配置。
- 校验精灵、性格、技能是否存在。
- 根据当前面板属性公式计算 `final_stats_json`。
- 保存技能槽顺序。
- 支持按 `elf_id` 查询已有配置。

### 2. 战斗阵容录入与启动流程

修改文件：

- `app/schemas/battle.py`
- `app/services/battle_service.py`
- `app/api/v1/endpoints/battles.py`

新增接口：

```http
POST /api/v1/battles/{battle_id}/lineup
POST /api/v1/battles/{battle_id}/start
POST /api/v1/battles/{battle_id}/switch
```

功能说明：

- 录入双方阵容。
- 己方精灵通过 `build_id` 复制确定配置。
- 敌方精灵只确认 `elf_id`，面板属性保持未知。
- 阵容录入时自动为敌方精灵生成候选配置。
- 启动战斗时确认双方首发。
- 切换精灵时执行 `clear_on_switch` 状态清除/保留规则。

### 3. 敌方候选配置落库

修改/新增文件：

- `app/services/candidate_service.py`
- `app/schemas/candidate.py`
- `app/api/v1/endpoints/candidates.py`

新增接口：

```http
POST /api/v1/candidates/{battle_id}/{elf_id}/generate
GET  /api/v1/candidates/{battle_id}/{elf_id}/summary
GET  /api/v1/candidates/{battle_id}/{elf_id}
```

功能说明：

- 读取数据库中的 `nature_definition`。
- 枚举个体资质分布。
- 计算候选面板属性。
- 写入 `build_candidate`。
- 技能组暂不笛卡尔展开，只保存可学习技能池。
- 提供候选数量、未排除数量、速度范围、最高置信度等摘要。

当前示例规则下，每只敌方精灵会生成 46320 个候选。

### 4. 统一状态实例管理

修改/新增文件：

- `app/schemas/effect.py`
- `app/services/effect_service.py`
- `app/api/v1/endpoints/effects.py`

新增接口：

```http
POST   /api/v1/effects/instances
DELETE /api/v1/effects/instances/{instance_id}
```

功能说明：

- 支持手动施加状态。
- 支持手动移除状态。
- 支持重复施加时按 `stack_rule` 处理。
- 支持切换精灵时按 `clear_on_switch` 自动清除或保留。
- 每次状态变化都会记录 `BattleEvent` 和 `EffectChangeEvent`。

### 5. 状态快照增强

修改文件：

- `app/services/snapshot_service.py`

功能说明：

- 快照保存全部生效状态实例 ID。
- 按 `owner_scope` 和阵营分组保存状态 ID。
- 保存完整状态实例副本。
- 同步记录当前双方上场精灵。
- 每次阵容录入、进入战斗、切换、伤害事件都可以生成快照。

### 6. 伤害事件记录

新增/修改文件：

- `app/schemas/event.py`
- `app/services/damage_event_service.py`
- `app/api/v1/endpoints/battles.py`

新增接口：

```http
POST /api/v1/battles/{battle_id}/damage-events
```

支持三种伤害输入：

| 类型 | 输入字段 | 处理方式 |
|---|---|---|
| `single_damage` | `damage_value` | 直接记录一次伤害 |
| `visual_total_damage` | `final_total_damage_value` | 按一次伤害事件记录最终总伤害 |
| `combo_repeated_damage` | `per_hit_damage_value` + `hit_count` | 系统计算 `computed_total_damage_value` |

伤害事件会：

- 创建通用 `BattleEvent`。
- 创建 `DamageEvent`。
- 创建事件快照。
- 更新受击方观测生命百分比。
- 构造 `DamageFormulaContext`。
- 调用推算引擎，但只返回公式未确认状态。

### 7. 计算占位层

新增/修改文件：

- `app/calculation/damage_calculator.py`
- `app/calculation/formula_context.py`
- `app/calculation/formula_registry.py`
- `app/calculation/errors.py`
- `app/calculation/speed_calculator.py`
- `app/inference/inference_engine.py`

功能说明：

- 定义 `DamageFormulaContext`。
- 定义统一的公式不可用返回结构。
- `DamageCalculator.calculate()` 当前固定返回：

```json
{
  "status": "formula_unavailable",
  "damage_value": null,
  "damage_range": null,
  "confidence": 0.0,
  "message": "伤害公式尚未确认，本次只记录事实，不执行候选排除。"
}
```

- `InferenceEngine.process_damage_event()` 当前不会排除候选，只返回占位推算结果。

### 8. 最小种子数据

修改文件：

- `app/seed/minimal_seed.py`

功能说明：

- 初始化数据库结构。
- 生成 30 个性格定义。
- 插入 2 个示例精灵。
- 插入 2 个示例技能。
- 插入 3 个示例状态：中毒、冰冻、示例天气。
- 插入精灵可学习技能关联。

运行命令：

```bash
python -m app.seed.minimal_seed
```

## 三、修改后的项目流程

### 1. 静态规则准备

项目启动前需要准备静态规则数据：

```text
ElfDefinition
NatureDefinition
SkillDefinition
EffectDefinition
ElfLearnableSkill
TypeEffectivenessRule
```

当前可以先运行：

```bash
python -m app.seed.minimal_seed
```

真实开发时应把示例数据替换为游戏规则数据。

### 2. 己方配置准备

玩家提前录入己方精灵配置：

```text
选择 elf_id
选择 nature_id
录入六维个体资质
选择技能组
系统计算 final_stats_json
保存 player_elf_build 和 player_elf_build_skill
```

### 3. 创建战斗

调用：

```http
POST /api/v1/battles
```

生成一场 `phase = preparation` 的战斗。

### 4. 录入双方阵容

调用：

```http
POST /api/v1/battles/{battle_id}/lineup
```

己方：

```json
{
  "side": "self",
  "elf_id": "elf_demo_fire",
  "build_id": "build_xxx",
  "is_active_elf": true
}
```

敌方：

```json
{
  "side": "enemy",
  "elf_id": "elf_demo_water",
  "is_active_elf": true
}
```

系统处理：

```text
己方 build -> BattleElfState
敌方 elf_id -> BattleElfState
敌方 elf_id -> BuildCandidate 批量生成
创建准备阶段状态快照
```

### 5. 进入战斗阶段

调用：

```http
POST /api/v1/battles/{battle_id}/start
```

系统处理：

```text
确认双方首发
phase = battle
创建状态快照
```

### 6. 战斗中记录事实

可记录：

```text
技能事件
伤害事件
状态施加
状态移除
精灵切换
生命百分比变化
```

伤害事件调用：

```http
POST /api/v1/battles/{battle_id}/damage-events
```

状态事件调用：

```http
POST /api/v1/effects/instances
DELETE /api/v1/effects/instances/{instance_id}
```

切换精灵调用：

```http
POST /api/v1/battles/{battle_id}/switch
```

### 7. 候选配置查看

调用：

```http
GET /api/v1/candidates/{battle_id}/{elf_id}/summary
```

当前只显示候选池状态：

```text
候选总数
未排除数量
已排除数量
速度区间
最高置信度
公式状态 formula_unavailable
```

### 8. 后续公式确认后的流程

未来公式确认后，伤害事件流程会变为：

```text
读取 DamageEvent
读取 BattleEffectSnapshot
构造 DamageFormulaContext
枚举 BuildCandidate
计算理论伤害
对比实际伤害和扣血百分比
更新 match_score / confidence / is_excluded
输出解释
```

当前已经为这条路径预留了接口和数据结构。

## 四、下一步应该做什么

### 优先级 1：补真实静态数据导入

需要把示例数据替换为真实数据：

- 精灵定义。
- 可学习技能池。
- 技能基础信息。
- 技能命中/连击规则。
- 状态定义。
- 属性克制表。
- 常见配置权重。

建议新建：

```text
app/importers/
  import_elves.py
  import_skills.py
  import_effects.py
  import_type_rules.py
```

### 优先级 2：补前端或接口调试页面

当前后端已经可以手动录入完整流程。下一步应做一个简单页面或 Postman 集合，按流程调用：

```text
创建己方配置
创建战斗
录入阵容
开始战斗
记录状态
记录伤害
查看候选摘要
```

### 优先级 3：完善事件类型

目前伤害事件和状态事件已经可用，下一步应补：

- 技能使用事件专用接口。
- 能量变化事件接口。
- 治疗事件接口。
- 回合推进接口。
- 战斗结束接口。

### 优先级 4：候选技能收敛

当前候选只保存可学习技能池。下一步应实现：

```text
敌方使用技能 -> confirmed_skill_ids_json 增加该技能
候选 possible_skill_ids_json 不包含该技能 -> 排除或降权
常见技能组 -> 只影响 match_score，不直接排除冷门配置
```

这部分不依赖伤害公式，可以先做。

### 优先级 5：公式确认后接入真实推算

等伤害公式确认后，再实现：

- 普通伤害公式。
- 取整顺序。
- 属性克制倍率。
- 增伤/减伤顺序。
- 防御技能生效位置。
- 连击每段计算方式。
- 特殊技能公式处理器。

主要修改位置：

```text
app/calculation/damage_calculator.py
app/calculation/formula_context.py
app/inference/inference_engine.py
```

## 六、替换说明

你可以把压缩包解压后直接覆盖原项目目录。建议先备份原项目，然后执行：

```bash
pip install -e '.[dev]'
python -m app.seed.minimal_seed
pytest
uvicorn app.main:app --reload
```

如果你已经有旧数据库，建议开发阶段先删除旧 SQLite 数据库后重新初始化。原因是本次修改补了不少服务逻辑和最小种子数据，旧库可能缺少必要的 `nature_definition`、`elf_learnable_skill` 等数据。
