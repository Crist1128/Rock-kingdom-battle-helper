# 本次修改说明：归档战斗物理删除 404 诊断与修复

## 背景

前端执行：

```http
DELETE /api/v1/admin/battles/{battle_id}/purge?dry_run=false
```

时出现 `404 Not Found`。仅凭访问日志无法判断是“路由未注册”，还是“路由已命中但后端服务层找不到该战斗”。本次修改同时增强后端诊断和前端删除流程。

## 修改内容

### 后端

1. 增加路由注册测试：
   - `app/tests/test_admin_battle_routes.py`
   - 确认以下路由在 FastAPI 应用中注册：
     - `DELETE /api/v1/admin/battles/{battle_id}/purge`
     - `GET /api/v1/admin/battles/{battle_id}/purge-plan`
     - `DELETE /api/v1/admin/battles/purge-archived`

2. 增强 purge 相关错误响应：
   - 战斗不存在时返回结构化错误：`code = battle_not_found`
   - 战斗不可物理删除时返回结构化错误：`code = battle_not_purgeable`
   - 附带 `message` 和 `hint`，便于前端展示可读提示。

### 前端

1. 单场物理删除前先调用：

```http
GET /api/v1/admin/battles/{battle_id}/purge-plan
```

确认路由、战斗存在且 `can_purge = true` 后，再执行真正删除。

2. 如果单场删除返回 404，会自动刷新归档战斗列表，避免继续对过期列表项操作。

3. 错误提示支持显示后端结构化 `message` 和 `hint`。

## 使用建议

本地覆盖后建议执行：

```bash
cd backend
python -m pytest -q
python -m ruff check app alembic
```

```bash
cd frontend
npm run typecheck
npm run build
```

如果仍然出现 404，请打开 `http://127.0.0.1:8000/docs`，确认是否能看到 `admin-battles` 分组下的 purge 接口。如果看不到，说明后端仍在运行旧代码，需要重启 uvicorn 或确认覆盖目录是否正确。
