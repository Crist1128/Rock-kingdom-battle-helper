# 本次爬虫代码与数据库数据管线修改总结

## 1. 结论：建议另起后端内的离线数据管线模块

本次不建议把爬虫直接做成普通后端业务接口，也不建议让前端直接触发完整爬取和写库流程。

推荐架构：

```text
BWIKI 爬虫 / 已有 CSV
        ↓
清洗模块 cleaner.py
        ↓
人工审阅 data/cleaned/*.json
        ↓
导入模块 importer.py 写入 SQLite
        ↓
后端现有 /elves、/skills 等查询接口读取数据库
```

原因：

1. 爬虫依赖外部站点结构、网络状态、反爬策略，不应影响后端核心业务稳定性。
2. 静态规则数据更新是批量写库操作，必须支持 dry-run、人工审阅、版本号和回滚。
3. MVP 阶段不做图片识别，图片下载不应耦合到数据库更新流程。
4. 后续如需要管理端功能，可以在该离线模块稳定后，再增加受控 admin endpoint 或后台任务，但不应作为普通 API 暴露。

本次新增模块位置：

```text
backend/app/data_pipeline/rocom/
  ├── scraper.py      # 改造后的 BWIKI 爬虫，默认不下载图片
  ├── cleaner.py      # 原始 CSV/JSON 清洗为后端可导入 JSON
  ├── importer.py     # dry-run / commit 导入 SQLite
  └── README.md       # 数据管线使用说明
```

## 2. 对上传爬虫输出数据的理解

对本次上传的爬虫结果压缩包解压后，关键数据如下：

| 文件 | 行数 / 规模 | 作用 |
|---|---:|---|
| `sprites.csv` | 465 行，21 列 | 精灵基础信息、种族值、属性、特性、克制关系、进化链、技能池 |
| `skills.csv` | 469 行，6 列 | 去重后的全技能定义 |
| `urls.csv` | 1335 行 | 图片 URL 元数据，包括精灵、属性、技能、特性、克制图标 |
| `lineups.csv` | 169 行，41 列 | PVP 配队数据，可用于常见技能组、性格、天赋分布的统计权重 |
| `sprites.backup.csv` | 465 行 | `sprites.csv` 备份 |

主要数据特征：

- 精灵属性共有 18 类：普通、火、水、草、电、冰、翼、地、幽、恶、光、毒、武、机械、萌、虫、龙、幻。
- 技能类型有：物攻、魔攻、状态、防御。
- `sprites.csv` 中存在同编号多形态数据，因此不能只用 `no` 做数据库主键。
- `skills` 字符串格式稳定，形如：`技能名(LV等级/属性/类型/威力/耗能/描述)`，可以可靠拆分为精灵可学习技能关系。
- `lineups.csv` 中的配队数据可以补充 `common_skill_sets_json`、`common_natures_json`、`common_individual_talent_patterns_json`，但只能作为候选权重参考，不应作为排除规则。

## 3. 本次清洗策略

### 3.1 精灵 ID

原始 `no` 存在重复，因此本次生成稳定 ID：

```text
rocom_elf_{编号}_{hash}
```

例如：

```text
rocom_elf_0001_8cceb6
```

这样可以保证同编号不同形态也能唯一入库。

### 3.2 技能 ID

技能使用技能名 hash 生成：

```text
rocom_skill_{技能名hash}
```

这样可以避免中文名称直接作为主键带来的迁移和引用问题，同时保持跨次导入稳定。

### 3.3 属性映射

中文属性会转换为后端内部枚举风格：

| 原始中文 | 入库值 |
|---|---|
| 普通 | `normal` |
| 火 | `fire` |
| 水 | `water` |
| 草 | `grass` |
| 电 | `electric` |
| 幽 | `ghost` |
| 恶 | `dark` |
| 光 | `light` |
| 武 | `fighting` |
| 机械 | `mechanical` |
| 幻 | `illusion` |

其他属性也已在 `cleaner.py` 中统一映射。

### 3.4 技能类型映射

| 原始类型 | 入库 `skill_category` |
|---|---|
| 物攻 | `physical` |
| 魔攻 | `magic` |
| 状态 | `status` |
| 防御 | `status` |

后端当前枚举没有 `defense`，所以防御技能先按状态技能入库，并在 `tags_json` 中保留 `defense` 标签。

### 3.5 伤害规则处理

MVP 阶段仍不确认完整伤害公式，因此本次没有根据描述臆造公式。

- 物攻 / 魔攻技能写入：`formula_unavailable`。
- 状态 / 防御技能不写伤害规则，描述保留在 `effect_operations_json` 的 `unparsed` 占位结构中。
- `hit_rule_json` 默认设置为 `single_damage`，并标记 `needs_manual_review=true`。

后续确认公式后，可以基于这些占位数据逐步补充规则。

### 3.6 属性克制规则

`sprites.csv` 中每只精灵都有页面克制关系，但双属性精灵的关系是复合结果，不能直接反推单属性矩阵。

本次策略：

- 单属性精灵页面：用于保守推导 `type_effectiveness_rule`。
- 双属性精灵页面：只保留在 `forms_json.type_matchup`，不用于反推全局克制矩阵。

基于本次上传数据，清洗出 113 条属性克制规则。

### 3.7 图片处理

MVP 阶段默认不下载图片。

- `scraper.py` 默认只记录图片 URL，不下载图片文件。
- 如需下载，显式加 `--with-images`。
- `cleaner.py` 默认使用 `urls.csv` 中的远程 URL 作为 `avatar` / 图片元数据引用。
- 后续接入图片识别时，可以切换 `--image-mode local` 并配合 `--with-images`。

## 4. 新增命令

以下命令均在 `backend` 目录执行。

### 4.1 安装爬虫依赖

```bash
pip install -e '.[crawler]'
```

本次在 `pyproject.toml` 中新增了可选依赖组：

```toml
crawler = [
  "requests>=2.32.0",
  "beautifulsoup4>=4.12.0",
]
```

### 4.2 从 BWIKI 重新爬取并清洗

```bash
python -m app.data_pipeline.rocom.scraper \
  --output ../data/raw/sprites_raw.json \
  --clean-output-dir ../data/cleaned \
  --data-version rocom_bwiki_20260516
```

默认不下载图片。

如需下载图片：

```bash
python -m app.data_pipeline.rocom.scraper \
  --output ../data/raw/sprites_raw.json \
  --clean-output-dir ../data/cleaned \
  --with-images
```

### 4.3 只清洗已有 CSV

```bash
python -m app.data_pipeline.rocom.cleaner \
  --sprites-csv ../data/sprites.csv \
  --skills-csv ../data/skills.csv \
  --urls-csv ../data/urls.csv \
  --lineups-csv ../data/lineups.csv \
  --output-dir ../data/cleaned \
  --data-version rocom_bwiki_20260516
```

清洗输出：

```text
../data/cleaned/elves.json
../data/cleaned/skills.json
../data/cleaned/elf_learnable_skills.json
../data/cleaned/type_effectiveness_rules.json
../data/cleaned/import_summary.json
```

### 4.4 Dry-run 导入数据库

```bash
python -m app.data_pipeline.rocom.importer \
  --cleaned-dir ../data/cleaned
```

默认 rollback，只打印新增 / 更新数量。

### 4.5 确认后提交数据库

```bash
python -m app.data_pipeline.rocom.importer \
  --cleaned-dir ../data/cleaned \
  --commit
```

## 7. 后续建议

1. 先使用 `cleaner.py` 生成 JSON，人工审阅 `import_summary.json` 和前几条 `elves.json`、`skills.json`。
2. 用 `importer.py` 默认 dry-run 检查写库数量是否符合预期。
3. 确认无误后再加 `--commit`。
4. 后续如果要做管理端更新功能，建议只让管理端触发 importer 的受控任务，不要让前端触发实时爬虫。
5. 图片识别模块启动前，再启用 `--with-images`，并把图片本地路径纳入 `recognition_templates_json` 或独立资源表。
