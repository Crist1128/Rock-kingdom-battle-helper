## 0. 字段速查

本节用于快速理解后文表格中的代码字段。审阅时可以先看中文含义，不需要记住具体数据库字段。

### 0.1 状态定义主字段

| 字段 | 中文含义 | 常见取值 | 审阅重点 |
|---|---|---|---|
| `effect_id` | 状态唯一 ID | `effect_burn`、`weather_rain` | 后续导入和代码引用都靠它，命名要稳定 |
| `effect_name` | 状态显示名称 | 灼烧、星陨印记、雨天 | 给玩家和前端看的中文名 |
| `category` | 状态分类标签 | `abnormal`、`mark`、`weather`、`stat_modifier`、`skill_modifier`、`damage_modifier`、`action_modifier`、`special_status` | 只用于分类和检索，不单独决定规则 |
| `polarity` | 正负面属性 | `positive`、`negative`、`neutral`、`mixed` | 用于展示、驱散增益/减益和筛选 |
| `display_group` | UI 展示分组 | `abnormal`、`mark`、`weather`、`buff`、`debuff`、`skill_slot` | 决定前端放在哪个状态区；印记底层挂队伍侧，但 UI 上与状态同栏展示 |
| `display_priority` | 展示排序 | 数字 | 数字越小或越大如何排序可由前端约定，核心是保持一致 |
| `owner_scope` | 状态挂载范围 | `elf`、`side`、`field`、`skill_slot`、`turn` | 最关键字段之一，决定状态跟随谁存在 |
| `target_scope` | 效果作用范围 | `single_elf`、`side`、`field`、`skill_slot` | 描述这个状态影响谁 |
| `attach_target_type` | 实例附着对象类型 | `elf`、`side`、`field`、`skill_slot` | 和 `owner_scope` 配合，决定运行时实例如何定位 |

### 0.2 `owner_scope` 怎么理解

| 取值 | 含义 | 例子 | 切换精灵时 |
|---|---|---|---|
| `elf` | 挂在某只精灵身上 | 灼烧、中毒、冻结、萌化 | 看 `clear_on_switch` 决定清除或保留 |
| `side` | 挂在某一方队伍侧 | 星陨印记、棘刺印记、奉献 | 通常不因当前精灵切换清除；印记虽然挂队伍侧，但 UI 上与状态同栏展示 |
| `field` | 挂在全场战场 | 雨天、沙暴、暴风雪 | 不因切换清除，通常被新天气替换 |
| `skill_slot` | 挂在某个技能槽 | 本技能威力永久+20、本技能能耗永久-1 | 跟随技能槽或本场技能状态；通常不在状态 UI 显示，而是在技能 UI 直接体现最终威力、能耗等 |
| `turn` | 只属于当前回合或一次行动窗口 | 本次先手、即时打断 | 通常回合结束或动作结束清除 |

### 0.3 层数、持续时间与清除字段

| 字段 | 中文含义 | 常见取值 | 审阅重点 |
|---|---|---|---|
| `default_layers` | 默认层数 | `1` | 技能未写层数时使用 |
| `max_layers` | 最大层数 | 数字或空 | 不知道上限时先留空并备注待确认 |
| `stack_rule` | 再次获得时如何处理层数 | `replace`、`add_layers`、`max`、`refresh`、`consume_uses` | 印记和异常通常要重点确认 |
| `refresh_rule` | 再次获得时是否刷新持续时间 | `refresh_duration`、`keep_existing` | 持续回合类状态需要确认 |
| `duration_type` | 持续类型 | `until_removed`、`turns`、`uses`、`permanent_in_battle` | 区分“持续 3 回合”“使用 1 次”“本场永久” |
| `default_duration_turns` | 默认持续回合数 | `3`、`7`、`8` | 天气、无法更换等需要明确 |
| `default_duration_uses` | 默认持续次数 | `1`、`2` | 奉献、下次技能、次数型效果可能使用 |
| `clear_on_switch` | 当前精灵切换时是否清除 | `true` / `false` | 灼烧/中毒通常清除；冻结/萌化/印记通常不清除 |
| `clear_by_abnormal_cleanse` | 是否可被清除异常去除 | `true` / `false` | 灼烧、中毒、寄生等异常通常为 true |
| `clear_by_stat_clear` | 是否可被清除属性变化去除 | `true` / `false` | 普通属性增减益通常为 true |
| `clear_by_mark_clear` | 是否可被清除印记去除 | `true` / `false` | 印记通常为 true |
| `clear_by_weather_replace` | 是否被新天气替换 | `true` / `false` | 天气通常为 true |
| `clear_by_skill_specific` | 是否需要特定技能规则处理 | `true` / `false` | 萌化、特殊印记、特殊转换类状态常用 |

### 0.4 特殊行为字段

| 字段 | 中文含义 | 用途 |
|---|---|---|
| `can_be_transferred` | 能否被转移 | 例如萌化转移、印记转移 |
| `can_be_converted` | 能否被转换 | 例如中毒转中毒印记、增益转中毒 |
| `can_be_inherited` | 能否被继承 | 预留给切换、队伍侧或特殊技能继承 |
| `can_be_stolen` | 能否被偷取 | 例如翅刃偷取印记 |
| `can_be_doubled` | 能否被翻倍 | 例如星陨印记翻倍、冻结翻倍、灼烧翻倍 |
| `conflict_group` | 互斥组 | 天气可统一使用 `weather` |
| `conflict_policy` | 冲突处理 | 天气通常是 `replace_old`，即新天气替换旧天气 |

### 0.5 参与计算的 JSON 字段

| 字段 | 中文含义 | 放什么规则 | 例子 |
|---|---|---|---|
| `formula_hooks_json` | 公式钩子列表 | 告诉计算器这个状态参与哪些环节 | `end_turn_status_damage`、`weather_multiplier`、`damage_reduction` |
| `stat_modifier_json` | 属性修正规则 | 物攻、魔攻、物防、魔防、速度等变化 | 物攻+100%、速度+80、双防-70% |
| `damage_modifier_json` | 伤害修正规则 | 吸血、增伤、减免转回复等；普通防御减伤建议独立走防御/应对窗口规则 | 吸血50%、减免转回复 |
| `skill_modifier_json` | 技能修正规则 | 技能威力、能耗、连击数、使用次数、冷却 | 全技能威力+40、本技能能耗永久-1 |
| `action_modifier_json` | 行动修正规则 | 先手、蓄力、眩晕、无法更换、打断 | 先手+1、下回合眩晕、3回合无法更换 |
| `resource_modifier_json` | 生命/能量结算规则 | 持续伤害、回复、能量变化 | 灼烧回合末扣血、寄生扣血并回血 |
| `special_rule_id` | 特殊规则处理器 ID | 无法用通用字段表达的规则 | 星陨伤害、奉献、萌化、无畏之心 |
| `developer_notes` | 开发备注 | 记录待确认点、来源技能和实现边界 | “第一阶段只记录，不参与硬排除” |

### 0.6 运行时相关名词

| 名词 | 含义 |
|---|---|
| `EffectDefinition` | 静态状态定义，说明“这个状态是什么、怎么结算、能否清除”。 |
| `BattleEffectInstance` | 单场战斗里的状态实例，说明“谁身上现在有这个状态、有几层、还剩多久”。 |
| `BattleEffectSnapshot` | 某个事件发生时的状态快照，用于历史伤害重算和候选证据链。 |
| `EffectOperation` | 技能产生的操作，例如施加状态、驱散状态、转化状态、回复生命、修改能量。 |

## 1. 建模边界

### 1.1 应进入 `EffectDefinition` 的内容

满足以下任一条件的效果，应统一进入 `EffectDefinition`，运行时统一生成 `BattleEffectInstance`：

1. 会在状态栏、印记栏、天气栏、技能槽或战斗 UI 中持续存在。
2. 会影响后续伤害、速度、连击、能耗、威力、行动顺序或切换判断。
3. 需要被状态快照记录，后续用于事件重放、伤害解释或候选推算。
4. 可以被驱散、转换、转移、偷取、继承、翻倍、切换清除或天气替换。

### 1.2 不建议单独建状态定义的内容

以下内容更适合作为 `SkillDefinition.effect_operations_json`、`damage_rule_json`、`hit_rule_json` 或 `special_rule_id`：

| 内容 | 建议处理 |
|---|---|
| 立即回复生命、能量变化 | `EffectOperation.heal` / `energy_change` |
| 立即驱散、转换、转移、偷取 | `EffectOperation.dispel_effect` / `convert_effect` / `transfer_effect` |
| 单次“本次技能威力翻倍” | 技能条件分支，不建长期状态 |
| 应对成功触发的即时打断 | `action_modifier` 或事件操作，按是否持续到下回合决定是否建实例 |
| 迸发、传动这类技能机制标签 | 迸发作为部分电系技能自带内容，优先放技能规则；传动暂不处理 |

## 2. 推荐分类

| category | 用途 | 典型 owner_scope | 例子 |
|---|---|---|---|
| `abnormal` | 异常与持续异常结算 | `elf` | 灼烧、中毒、寄生、冻结 |
| `mark` | 印记 | `side` | 星陨印记、棘刺印记、湿润印记；底层跟随队伍侧，UI 上与状态同栏 |
| `weather` | 全场天气 | `field` | 雨天、沙暴、暴风雪 |
| `stat_modifier` | 六维属性增减益 | `elf` | 物攻+100%、双防-70%、速度+80 |
| `skill_modifier` | 技能威力、能耗、连击、使用次数、冷却等修正 | `elf`，少数为 `skill_slot` | 全技能威力+40、本技能能耗永久-1；技能槽修正不在状态 UI 显示，直接体现在技能 UI |
| `damage_modifier` | 吸血、增伤等伤害链修正 | `elf` | 吸血50%；普通防御减伤不作为常规状态 |
| `action_modifier` | 行动、先手、蓄力、眩晕、无法更换等 | `elf` 或 `turn` | 迅捷、蓄力、眩晕、无法更换精灵 |
| `special_status` | 有专属规则但需要记录的状态 | `elf` 或 `side` | 萌化；奉献暂不讨论 |

字段原则：

- `category` 只做分类标签，实际规则由 `formula_hooks_json`、`*_modifier_json`、`clear_by_*`、`special_rule_id` 决定。
- 历史伤害解释必须读取 `BattleEffectSnapshot.full_snapshot_json`，不能读取当前状态代替历史状态。
- 未确认公式或持续规则的效果可以先导入为可显示状态，但 `developer_notes` 必须写明“待确认”，计算器遇到时输出 `unknown_factors`。
- 防御技能产生的减伤建议作为“防御/应对窗口”独立处理，由技能规则解析到伤害公式上下文，不作为普通可选状态批量导入。

印记通用字段建议：

- `category = mark`
- `display_group = mark`，但前端展示时应和常规状态同栏或同区域展示；不要因为底层 `owner_scope = side` 就把印记藏到队伍信息里。
- `clear_on_switch = false`
- `clear_by_mark_clear = true`
- `can_be_transferred = true` 仅对确认可转移/偷取的印记开启；当前至少 `翅刃` 存在“偷取印记”分支。
- `can_be_doubled = true` 仅对确认可翻倍的印记开启；当前星陨存在翻倍分支。
- `developer_notes` 记录来源技能与未确认结算。

## 5. 天气候选清单

天气统一作为 `BattleEffectInstance`，不建立平行天气系统。

| 建议 effect_id | 名称 | category | polarity | owner_scope | duration_type | 默认持续 | clear_on_switch | conflict_group | 来源技能 | 已知行为 |
|---|---|---|---|---|---|---:|---:|---|---|---|
| `weather_rain` | 雨天 | `weather` | `neutral` | `field` | `turns` | 7 | false | `weather` | 落雨 | 水系技能增伤 `75%` |
| `weather_sandstorm` | 沙暴 | `weather` | `neutral` | `field` | `turns` | 7 | false | `weather` | 沙涌 | 地系技能能耗减半，向下取整 |
| `weather_blizzard` | 暴风雪 / 冬至天气 | `weather` | `neutral` | `field` | `turns` | 7 | false | `weather` | 冬至 | 双方队伍每回合结束获得 `2` 层冻结 |

天气通用字段建议：

- `clear_by_weather_replace = true`
- `conflict_policy = replace_old`
- `formula_hooks_json` 按天气效果分别写入：雨天为 `["weather_damage_bonus"]`，沙暴为 `["weather_skill_cost_modifier"]`，暴风雪/冬至天气为 `["end_turn_apply_effect"]`。

## 7. 普通属性增益/减益

- 普通属性增益/减益最多 `99` 层。
- 物攻、魔攻、物防、魔防等百分比属性每层 `10%`。

## 8. 技能修正类状态

## 9. 伤害修正与防御机制

减伤更接近“防御/应对窗口”机制，而不是常规状态。它表达的是“对方本次攻击被某个防御技能减免”，通常只作用于本次攻击或当前应对窗口，不应作为普通可选状态进入状态栏。

### 9.1 防御减伤放在哪里处理

建议处理位置：

1. `SkillDefinition.damage_rule_json` 或 `effect_operations_json` 记录防御技能本身的减伤比例、应对条件和附加操作。
2. `RuleResolver` 在解析本次行动时输出 `damage_reductions`，例如 `[{ "source_skill": "防御", "value": 0.70 }]`。
3. `DamageFormulaContext` 保存本次攻击实际生效的减伤列表，用于伤害计算和解释链。
4. `BattleEffectSnapshot` 仍然保存事件发生时的状态事实，但不需要为了普通防御减伤额外创建长期 `BattleEffectInstance`。

如果某个减伤明确会跨回合持续或显示为状态图标，再单独建 `EffectDefinition`；普通防御技能的 `减伤70%` 不作为状态定义批量导入。

### 9.2 仍需进入状态系统的伤害修正

| 建议 effect_id | 名称 | category | owner_scope | 已出现取值 | 来源示例 | 建议 |
|---|---|---|---|---|---|---|
| `lifesteal_modifier` | 吸血 | `damage_modifier` | `elf` | `50%`、`100%`、奉献 `10%` | 贪婪、等价交换、暗突袭、龙之利爪、虫茧 | 跟随精灵；按造成伤害后的二级事件回复 |
| `damage_to_heal_modifier` | 减免转回复 | `damage_modifier` 或 `special_status` | `turn` | 减免伤害转生命回复 | 无畏之心 | 特殊规则，需 `special_rule_id` |

说明：

- 吸血是跟随 `elf` 的状态或修正。
- 多个减伤按伤害设计文档使用 `product(1 - reduction_i)`，不要简单相加；但这些减伤来自防御/应对窗口上下文，而不是普通状态列表。

## 13. 待审阅问题

以下问题会影响导入字段和后续计算，建议在进入导入实现前确认：

1. 萌化需要精灵形态链与各形态种族值数据，否则只能记录层数，不能正确改变面板。
2. 萌化印记的来源技能仍需补充，目前只确认效果为“获得增益时额外获得 1 层”。
3. 奉献暂不讨论，后续单独确认是“层数”还是“次数队列”。
4. “永久”修正的生命周期：是本场战斗永久、跟随技能槽、跟随精灵，还是切换后保留但出场重置。
5. 防御/应对窗口的数据结构还需要在实现时落到 `RuleResolver` 和 `DamageFormulaContext`，确保减伤能进入解释链但不污染普通状态表。
6. `打断` 是否有持续图标；如果只是即时打断，应只作为事件操作，不导入持久状态。
7. 状态图标识别素材尚未补齐，`is_recognizable_by_icon` 第一阶段应默认 `false`，等图像识别阶段再更新。

---

## 15. 本文结论

1. 第二步补全状态定义可以先从 P0/P1 入手，不需要等所有技能 DSL 完成。
2. 核心异常与核心印记已有足够口径，可以进入第一版 `effect_definition`。
3. 大量增益/减益不应按“每个数值一条定义”硬编码，建议以参数化状态族 + 运行时 payload 表达。
4. 驱散、转化、偷取、翻倍、应对、迸发、传动等不要混入普通状态清单，应在技能规则操作或特殊规则中处理。
5. 所有待确认状态可以先导入为可选可见状态，但计算器必须把它们视为未知因素，不能用于候选硬排除。
