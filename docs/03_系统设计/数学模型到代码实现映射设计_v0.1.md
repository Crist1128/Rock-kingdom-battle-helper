## 1. 总体原则

### 1.1 分层原则

数学模型落到代码时，建议拆成四层：

```text
规则数据层
  SkillDefinition / EffectDefinition / TypeEffectivenessRule
        ↓
规则解析层
  RuleResolver / ModifierResolver / TypeEffectivenessResolver
        ↓
纯计算层
  StatCalculator / DisplayPowerCalculator / AttackDamageCalculator / StatusDamageCalculator
        ↓
推算应用层
  DamageEventService / InferenceEngine / CandidateService
```

核心原则：

1. **项目主线是敌方配置反推，公式层服务于观测解释和候选评分。**
2. **公式层只算数值，不判断复杂业务分支。**
3. **应对成功、状态触发、防御技能效果由规则解析层解析。**
4. **历史计算必须读取事件发生时的快照。**
5. **候选过滤初期只软评分，不硬排除。**
6. **规则 JSON 必须结构化，不能只保存自然语言描述。**

### 1.2 当前不建议立即新建的表

短期不建议立即新建：

```text
skill_conditional_rule
formula_rule
status_settlement_rule
```

理由：

- 现有 `SkillDefinition.damage_rule_json`、`hit_rule_json`、`effect_operations_json` 足够承载第一版规则；
- `EffectDefinition.formula_hooks_json`、`damage_modifier_json`、`resource_modifier_json` 足够承载第一版状态规则；
- 规则尚未稳定，过早建表会造成频繁迁移。

建议：**先用结构化 JSON + dry-run 校验；规则稳定后再拆表。**

## 4. 候选生成代码实现映射

当前实现位置：

```text

```

当前候选生成流程：

```text
generate_for_enemy_elf(battle_id, elf_id)
  -> 读取 ElfDefinition

  -> 生成 IndividualTalentDistribution 集合
  -> 读取 ElfLearnableSkill 技能池
  -> 计算 PanelStats
  -> 写入 BuildCandidate
```

数学对应：

$$
\mathcal{C}_e=\{(e,n,\mathbf{I},\mathbf{S},\mathcal{K}) : n\in\Omega_N,\mathbf{I}\in\Omega_I\}
$$

字段写入：

| 数学项 | 写入字段 |
|---|---|
| $n$ | `BuildCandidate.nature_id` |
| $\mathbf{I}$ | `BuildCandidate.individual_talent_distribution_json` |
| $\mathbf{S}$ | `BuildCandidate.final_*` |
| $\mathcal{K}$ | `BuildCandidate.possible_skill_ids_json` |
| 已确认技能 | `BuildCandidate.confirmed_skill_ids_json` |
| 初始评分 | `match_score = 0.0` |
| 初始置信度 | `confidence = 0.0` |
| 初始排除状态 | `is_excluded = False` |

### 4.1 候选生成建议改进

#### 4.1.1 候选数量校验

生成后应校验：

$$
|\mathcal{C}_e|=30\times1544=46320
$$

如果性格数量不是 30，应记录 warning，因为后续候选数量会变化。

#### 4.1.2 技能池缺失策略

如果 `possible_skill_ids_json` 为空，不建议后续根据技能事件硬排除候选。应标记：

```json
{
  "unknown_factors": ["learnable_skill_pool_empty"]
}
```

## 5. 技能规则 JSON 映射

现有字段：

```text
SkillDefinition.damage_rule_json
SkillDefinition.hit_rule_json
SkillDefinition.effect_operations_json
```

建议第一阶段不要新建表，而是使用结构化 JSON。

### 5.1 `damage_rule_json`

推荐结构：

```json
{
  "damage_type": "normal_formula",
  "offense_category": "physical",
  "power_source": "base_power",
  "base_power_override": null,
  "affected_by_type_effectiveness": true,
  "affected_by_stab": true,
  "affected_by_weather": true,
  "affected_by_reduction": true,
  "unstable_modifiers": [],
  "special_formula_id": null,
  "conditional_effects": [
    {
      "branch_id": "response_success",
      "trigger": "response_success",
      "conditions": {
        "manual_required_when_unknown": true
      },
      "operations": [
        {
          "type": "modify_response_multiplier",
          "value": 3.0
        }
      ]
    }
  ],
  "needs_manual_review": true
}
```

### 5.2 字段与数学项映射

| 数学项 | JSON 字段 | 说明 |
|---|---|---|
| $P_0$ | `base_power` 或 `base_power_override` | 默认用 `SkillDefinition.base_power` |
| $R_{resp}$ | `conditional_effects.operations.modify_response_multiplier` | 由 `RuleResolver` 输出 |
| $R_{unstable}$ | `unstable_modifiers` | 未确认时进入未知因素 |
| 是否吃克制 | `affected_by_type_effectiveness` | false 时 $R_{type}=1$ |
| 是否吃本系 | `affected_by_stab` | false 时 $R_{stab}=1$ |
| 是否吃天气 | `affected_by_weather` | false 时 $R_{weather}=1$ |
| 是否吃减伤 | `affected_by_reduction` | false 时 $R_{reduce}=1$ |
| 特殊公式 | `special_formula_id` | 分发到特殊处理器 |

### 5.3 `hit_rule_json`

推荐结构：

```json
{
  "damage_display_type": "combo_repeated_damage",
  "is_combo": true,
  "hit_count_type": "fixed",
  "base_hit_count": 3,
  "per_hit_same_damage": true,
  "combo_calculation_mode": "per_hit_formula_then_repeat",
  "combo_modifier_hooks": [],
  "needs_manual_review": false
}
```

字段与数学项：

| 数学项 | JSON 字段 |
|---|---|
| 是否连击 | `is_combo` |
| 连击数 $h$ | `base_hit_count` + 规则修正 |
| 单段是否相同 | `per_hit_same_damage` |
| 连击计算方式 | `combo_calculation_mode` |

### 5.4 `effect_operations_json`

用于技能造成的状态变化，不建议混入纯伤害公式：

```json
[
  {
    "condition": "response_success",
    "operation": "apply_effect",
    "effect_id": "effect_burn",
    "target": "defender",
    "layers": 2
  }
]
```

`RuleResolver` 负责筛选 condition，`EffectEngine` 或 `BattleEffectService` 负责执行。

## 6. 状态规则 JSON 映射

现有字段：

```text
EffectDefinition.formula_hooks_json

EffectDefinition.skill_modifier_json
```

### 6.1 `formula_hooks_json`

用于声明状态参与哪个计算环节：

```json
[
  "stat_stage_attack_up",
  "damage_reduction",
  "end_turn_status_damage"
]
```

建议 hook 枚举：

| Hook | 用途 |
|---|---|
| `stat_stage_attack_up` | 攻击提升，参与 $R_{stage}$ |
| `stat_stage_attack_down` | 攻击降低 |
| `stat_stage_defense_up` | 防御提升 |
| `stat_stage_defense_down` | 防御降低 |
| `display_power_multiplier` | 威力提升增益 |
| `flat_power_bonus` | 固定威力加成 |
| `damage_reduction` | 减伤 |
| `weather_multiplier` | 天气倍率 |
| `end_turn_status_damage` | 回合末状态伤害 |
| `switch_in_damage` | 上场触发伤害 |
| `post_attack_trigger_damage` | 攻击后触发伤害，如星陨 |

### 6.2 `stat_modifier_json`

属性等级修正：

```json
{
  "target_stat": "physical_attack",
  "direction": "up",
  "value": 1.0,
  "applies_to_skill_category": ["physical"]
}
```

映射到：

$$
R_{stage}=\frac{1+u_A+d_D}{1+d_A+u_D}
$$

### 6.3 `damage_modifier_json`

减伤、增伤、不稳定加成：

```json
{
  "modifier_type": "damage_reduction",
  "value": 0.5,
  "stack_mode": "multiply",
  "applies_to": {
    "formula_type": ["attack", "starfall"],
    "skill_categories": ["physical", "magic"]
  }
}
```

映射到：

$$
R_{reduce}=\prod_i(1-q_i)
$$

### 6.4 `resource_modifier_json`

状态百分比伤害、回血等：

```json
{
  "settlement_type": "end_turn",
  "damage_kind": "status",
  "percent_of": "target_max_hp",
  "percent_per_layer": 0.02,
  "element_type": "fire",
  "uses_type_effectiveness": true,
  "affected_by_reduction": false,
  "heal_opponent_by_damage": false,
  "after_settlement": {
    "layer_change": "halve_floor"
  }
}
```

映射到：

$$
D_{status}=\left\lfloor S_{HP}^{max}\cdot p\cdot L\cdot R_t\right\rfloor
$$

```python
def calculate_display_power(context: DamageFormulaContext) -> Decimal:
    return (
        (context.base_power * context.response_multiplier + context.flat_power_bonus)
        * context.power_multiplier
        * context.stat_stage_multiplier
        * context.stab_multiplier
```

### 9.3 攻击伤害函数

文件：

```text
backend/app/calculation/attack_damage.py
```

关键函数：

```python
def calculate_attack_damage(context: DamageFormulaContext) -> DamageCalculationResult:
    offense, defense = select_offense_defense_stats(context)
    display_power = calculate_display_power(context)
    reduction_product = product(Decimal("1") - item for item in context.damage_reductions)
    unstable = context.unstable_multiplier

    if unstable is None:
        return partial_range_result(...)

    raw = (offense / defense) * Decimal(37) / Decimal(41)
    raw *= display_power * unstable * reduction_product
    damage = floor_damage(raw)
    return DamageCalculationResult(...)
```

对应公式：

$$
D_{attack}=\left\lfloor\left(\frac{A}{D}\cdot\frac{37}{41}\right)\cdot(P_{display}\cdot R_{unstable})\cdot R_{reduce}\right\rfloor
$$

连击：

```python
if context.hit_count and context.hit_count > 1:
    per_hit_damage = damage
    total_damage = per_hit_damage * context.hit_count
```

### 9.6 星陨伤害函数

文件：

```text
backend/app/calculation/starfall_damage.py
```

建议结果中显式记录“触发技能类别”和“星陨结算属性”两类信息：

```python
class StarfallDamageContext(BaseModel):
    trigger_skill_id: str | None = None
    trigger_skill_category: str  # physical | magic，用于选择物攻/物防或魔攻/魔防
    trigger_skill_element_type: str  # 触发技能本身必须不是 illusion
    starfall_element_type: str = "illusion"  # 星陨结算伤害记录为幻系
    layers: int
```

星陨威力：

```python
def calculate_starfall_power(layers: int) -> int:
    return layers * layers + 24 * layers - 24
```

攻防属性选择跟随触发技能类别：

```python
def select_starfall_offense_defense(
    context: DamageFormulaContext,
    starfall_context: StarfallDamageContext,
) -> tuple[Decimal, Decimal]:
    if starfall_context.trigger_skill_category == "physical":
        return context.attacker_panel_stats.physical_attack, context.defender_panel_stats.physical_defense
    if starfall_context.trigger_skill_category == "magic":
        return context.attacker_panel_stats.magic_attack, context.defender_panel_stats.magic_defense
    raise UnsupportedFormulaError("starfall requires physical or magic trigger skill")
```

伤害计算：

```python
def calculate_starfall_damage(
    context: DamageFormulaContext,
    starfall_context: StarfallDamageContext,
) -> DamageCalculationResult:
    if starfall_context.trigger_skill_element_type == "illusion":
        raise UnsupportedFormulaError("illusion trigger skill does not trigger starfall")

    power = calculate_starfall_power(starfall_context.layers)
    offense, defense = select_starfall_offense_defense(context, starfall_context)
    reduction_product = product(Decimal("1") - item for item in context.damage_reductions)
    raw = (offense / defense) * Decimal(37) / Decimal(41) * power * reduction_product
    damage = floor_damage(raw)
    return DamageCalculationResult(
        status="calculated",
        formula_type="starfall",
        damage_value=damage,
        explanation={
            "starfall_element_type": starfall_context.starfall_element_type,
            "trigger_skill_category": starfall_context.trigger_skill_category,
            "trigger_skill_element_type": starfall_context.trigger_skill_element_type,
            "uses_display_power": False,
            "uses_stab": False,
            "uses_type_effectiveness": False,
            "uses_weather": False,
            "uses_unstable_multiplier": False,
        },
    )
```

实现注意：

1. 触发条件仍然是“非幻属性攻击技能命中或作用于防御方”。
2. 星陨结算伤害本身记录为幻系伤害。
3. 星陨公式不额外乘属性克制、本系、天气、显示威力或不稳定加成。
4. 星陨仍受防御技能减伤影响。
5. 星陨建议生成独立 `DamageEvent`，并在 `formula_context_json` 中记录 `parent_attack_event_id` / `trigger_skill_id`，避免和普通攻击伤害混成一个观测值。

## 11. 候选反推代码映射

当前位置：

```text

```

当前已接入 Observation API、普通攻击最小 `DamageCalculator` 和 `RuleResolver` 雏形。目标仍是把玩家录入的观测事件转化为候选评分，而不是只返回一次正向计算结果；完整状态/特殊公式未覆盖时应返回 unknown 或 `formula_unavailable`。

### 11.1 目标流程

```text
process_observation_event(event, context)
  -> 判断敌方在攻击方还是防御方
  -> 加载对应 BuildCandidate active rows
  -> 对每个 candidate 构造 candidate_context
  -> ObservationMatcher 根据事件类型分发
      -> DamageCalculator / StatusDamageCalculator / SpeedMatcher / SkillPoolMatcher
  -> 与玩家观测值比较
  -> 更新 match_score / confidence / matched_event_ids_json / mismatched_event_ids_json / evidence_ids_json
  -> 默认不硬排除，仅在满足硬排除条件且开关开启时设置 is_excluded
```

为了兼容当前实现，可以先让 `process_damage_event()` 内部调用更通用的 `process_observation_event()`。

## 15. 需要审阅确认的问题

以下实现口径已按数学建模文档收敛：

- 冻结生命比例在代码内部统一使用 `0~1`，前端或接口的 `0~100` 输入由 `ContextBuilder` 归一化。
- 星陨按攻击伤害分支实现，建议创建独立 `DamageEvent(formula_type=starfall)`，并用 `formula_context_json` 记录父攻击事件与触发技能。
- 星陨攻防属性跟随触发技能类别，结算属性为幻系，不乘显示威力、本系、克制、天气或不稳定加成。

编码前仍建议确认：

1. 是否接受短期继续使用 JSON 规则，而不是新建 `skill_conditional_rule` 表？
2. `DamageFormulaContext` 是直接扩展原类，还是新增 V2 类并兼容旧字段？
3. `RuleResolver` 的输出是否作为 `formula_context_json` 的一部分持久化？
4. 其他状态伤害应创建 `DamageEvent`，还是只创建 `ResourceChangeEvent`？
5. 是否先只实现攻击伤害，不自动结算状态？
6. 候选软评分的初始权重和惩罚系数如何设置？
7. 何时开启 `ENABLE_DAMAGE_HARD_FILTER`？
8. 是否需要为规则 JSON 写独立 JSON Schema 校验器？
9. 前端是否需要先增加手动输入 `response_success`、`manual_flags`？

## 16. 建议结论

建议第一版实现路线：

```text
不改数据库表
  -> 用现有 JSON 字段承载规则
  -> 新增 RuleResolver / ModifierResolver / TypeEffectivenessResolver
  -> 扩展 DamageFormulaContext
  -> 实现纯攻击伤害计算
  -> 在 DamageEventService 返回解释链
  -> 新增 ObservationMatcher，把玩家观测转为候选评分
  -> 候选只软评分并输出分布/证据链
  -> 状态自动结算后置
```

这样能最大限度复用当前项目结构，同时把 MVP 重心放在“玩家观测 -> 候选反推”上，并避免错误公式或不完整应对规则直接污染候选排除结果。
