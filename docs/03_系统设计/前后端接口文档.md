# Rock PVP Helper 前后端接口文档

> 审阅范围：当前前后端 MVP、Observation API、候选软评分与 RuleResolver 雏形。
>
> 后端基准地址：`http://127.0.0.1:8000/api/v1`  
> 前端开发地址：`http://127.0.0.1:5173`  
> 前端默认 API 基址：`/api/v1`，通过 Vite proxy 转发到后端。

---

## 1. 当前系统接口总览

### 1.1 前端页面与后端接口关系

| 前端页面 | 路由 | 主要后端接口 |
|---|---|---|
| 己方配置 | `/builds` | `/elves`、`/elves/{elf_id}/skills`、`/natures`、`/player-builds` |
| 战斗首页 | `/` | `/battles`、`/battles/{battle_id}/archive` |
| 准备阶段 | `/preparation` | `/battles`、`/player-builds`、`/battles/{battle_id}/lineup`、`/battles/{battle_id}/start` |
| 战斗工作台 | `/battle` | `/battles/{battle_id}/state`、`/battles/{battle_id}/switch`、`/battles/{battle_id}/finish`、`/effects`、`/effects/instances`、`/battles/{battle_id}/damage-events`、`/battles/{battle_id}/resource-events`、`/observations/{battle_id}`、`/candidates/*` |
| 事件日志 | `/events` | `/battles/{battle_id}/timeline`、`/battles/{battle_id}/events`、`/battles/{battle_id}/events/{event_id}/void`、`/battles/{battle_id}/events/{event_id}/correct`、`/battles/{battle_id}/replay-from/{event_id}` |
| 规则库 | `/rules` | `/elves`、`/skills`、`/natures`、`/effects` |
| 设置 | `/settings` | `/health`、`/admin/battles/*` |

### 1.2 当前接口状态

| 模块 | 状态 | 说明 |
|---|---:|---|
| 静态规则查询 | 可用 | 精灵、技能、性格、状态定义均可查询。 |
| 己方配置 | 可用 | 支持创建、编辑、技能槽排序、软删除。 |
| 战斗管理 | 可用 | 支持创建、列表、准备阶段阵容、开始、切换、结束、归档、状态查询。 |
| 状态系统 | 可用 | 支持状态定义筛选、状态实例施加与移除。 |
| 伤害事件 | 可用 | 支持单次、动画多段总伤害、连击单段伤害与次数；创建事件本身仍以记录事实为主。 |
| 观察事件 / 候选软评分 | 可用 | `POST /observations/{battle_id}` 支持伤害值、扣血百分比、技能出现、速度先后手等观测，更新候选评分、置信度和 evidence。 |
| 资源事件 | 可用 | 支持生命/能量手动变化记录。 |
| 候选配置 | 可用 | 支持候选生成、摘要、详情、分页明细和软评分结果展示；硬排除默认关闭。 |
| 事件纠错/重放 | 部分可用 | 支持作废、创建修正事件；重放接口目前是占位，不做真实重算。 |
| 归档清理 | 可用 | 支持归档战斗 dry-run 预览与物理删除，受管理令牌保护。 |
| 数据更新 | 可用 | 支持 rocom 检查、远程同步、本地导入、任务查询，受管理令牌保护。 |
| 伤害公式/速度公式/候选排除 | 部分实现 | 已有普通攻击最小公式、RuleResolver 雏形和基础速度观测匹配；完整状态/特殊公式、速度概率和候选硬排除仍未实现。 |

## 2. 通用约定

### 2.1 请求头

普通接口：

```http
Content-Type: application/json
```

管理接口如 `/admin/data-updates/*`、`/admin/battles/*`：如果后端配置了 `ADMIN_UPDATE_TOKEN`，需要额外传：

```http
X-Admin-Token: <你的管理令牌>
```

如果 `.env` 中 `ADMIN_UPDATE_TOKEN` 为空，则当前实现不会强制校验令牌。

### 2.2 常见枚举

#### 阵营 `side`

| 值 | 含义 |
|---|---|
| `self` | 己方 |
| `enemy` | 敌方 |

#### 战斗阶段 `phase`

| 值 | 含义 |
|---|---|
| `preparation` | 准备阶段 |
| `battle` | 战斗中 |
| `finished` | 已结束 |
| `archived` | 已归档 |

#### 伤害显示类型 `damage_display_type`

| 值 | 含义 | 必填字段 |
|---|---|---|
| `single_damage` | 单次伤害 | `damage_value` |
| `visual_total_damage` | 动画多段但最终显示总伤害 | `final_total_damage_value` |
| `combo_repeated_damage` | 连击，每段相同且无总伤害 | `per_hit_damage_value`、`hit_count` |
| `special_damage` | 特殊结算伤害 | 当前未专门实现公式逻辑 |

#### 状态归属范围 `owner_scope`

| 值 | 含义 | 施加状态时需要字段 |
|---|---|---|
| `elf` | 挂在某只精灵身上 | `owner_side`、`owner_elf_id` |
| `side` | 挂在某一队伍侧 | `owner_side` |
| `field` | 全战场 | 可用默认 `field_id=main` |
| `skill_slot` | 某个技能槽 | `owner_skill_slot_id` |
| `turn` | 当前回合临时状态 | 按后续规则扩展 |

## 4. 静态规则接口

## 4.1 精灵接口

### 4.1.1 查询精灵列表

```http
GET /api/v1/elves?q=<关键词>&limit=50&offset=0
```

Query 参数：

| 参数 | 类型 | 默认 | 说明 |
|---|---:|---:|---|
| `q` | string | 空 | 按 `elf_name` 模糊搜索 |
| `limit` | int | 50 | 1-500 |
| `offset` | int | 0 | 分页偏移 |

响应字段：

```ts
interface ElfDefinitionOut {
  elf_id: string;
  elf_name: string;
  avatar: string;
  element_types_json: string;
  base_hp_talent: number;
  base_physical_attack_talent: number;
  base_physical_defense_talent: number;
  base_magic_attack_talent: number;
  base_magic_defense_talent: number;
  base_speed_talent: number;
  data_version?: string | null;
}
```

前端调用：

```ts
api.elves.list({ q, limit, offset })
```

### 4.1.2 查询单只精灵

```http
GET /api/v1/elves/{elf_id}
```

前端调用：

```ts
api.elves.get(elfId)
```

### 4.1.3 查询精灵可学习技能

```http
GET /api/v1/elves/{elf_id}/skills?q=<关键词>&limit=100&offset=0
```

用途：己方配置页选择技能时，按当前精灵过滤技能池。

前端调用：

```ts
api.elves.skills(elfId, { q, limit, offset })
```

# 5. 己方配置接口

## 5.1 创建己方配置

```http
POST /api/v1/player-builds
```

请求体：

```json
{
  "elf_id": "rocom_elf_0001",
  "nature_id": "nature_xxx",
  "individual_talent_distribution": {
    "hp": 0,
    "physical_attack": 0,
    "physical_defense": 10,
    "magic_attack": 10,
    "magic_defense": 0,
    "speed": 10
  },
  "skill_ids": ["skill_a", "skill_b", "skill_c", "skill_d"],
  "build_name": "高速魔攻配置",
  "is_default": false,
  "notes": "备注"
}
```

校验规则：

| 字段 | 规则 |
|---|---|
| `individual_talent_distribution.*` | 0-10 |
| `skill_ids` | 最多 4 个，不能重复 |

响应字段：

```ts
interface PlayerElfBuildOut {
  build_id: string;
  build_name?: string | null;
  elf_id: string;
  elf_name?: string | null;
  avatar?: string | null;
  element_types_json?: string | null;
  nature_id: string;
  individual_talent_distribution_json: string;
  final_stats_json?: string | null;
  skill_ids: string[];
  is_default: boolean;
  notes?: string | null;
}
```

前端调用：

```ts
api.playerBuilds.create(payload)
```

## 5.2 查询己方配置列表

```http
GET /api/v1/player-builds?elf_id=<精灵ID>
```

`elf_id` 可选。不传则返回所有未软删除配置。

前端调用：

```ts
api.playerBuilds.list(elfId?)
```

## 5.3 查询单个己方配置

```http
GET /api/v1/player-builds/{build_id}
```

前端调用：

```ts
api.playerBuilds.get(buildId)
```

## 5.4 更新己方配置

```http
PUT /api/v1/player-builds/{build_id}
```

请求体同创建配置。当前采用整单替换策略，后端会重新计算面板属性并重建技能槽。

前端调用：

```ts

```

## 5.5 仅替换技能槽顺序

```http
PUT /api/v1/player-builds/{build_id}/skills
```

请求体：

```json
{
  "skill_ids": ["skill_a", "skill_b", "skill_c", "skill_d"]
}
```

前端调用：

```ts
api.playerBuilds.replaceSkills(buildId, skillIds)
```

## 5.6 删除己方配置

```http
DELETE /api/v1/player-builds/{build_id}
```

说明：软删除，不物理清理历史引用。

前端调用：

```ts
api.playerBuilds.delete(buildId)
```

# 6. 战斗管理接口

## 6.1 创建战斗

```http
POST /api/v1/battles
```

请求体：

```json
{
  "battle_name": "测试战斗",
  "notes": "备注"
}
```

响应字段：

```ts
interface BattleOut {
  battle_id: string;
  battle_name?: string | null;
  phase: "preparation" | "battle" | "finished" | "archived";
  turn_number: number;
  self_active_elf_id?: string | null;
  enemy_active_elf_id?: string | null;
  current_snapshot_id?: string | null;
  notes?: string | null;
  created_at?: string | null;
  updated_at?: string | null;
}
```

前端调用：

```ts
api.battles.create(payload)
```

## 6.2 查询战斗列表

```http
GET /api/v1/battles?phase=<阶段>&include_archived=false&limit=50&offset=0
```

说明：

- 默认不返回 `archived` 战斗。
- 需要查看归档记录时，传 `phase=archived` 或 `include_archived=true`。

前端调用：

```ts
api.battles.list({ phase, include_archived, limit, offset })
```

## 6.3 查询单场战斗

```http
GET /api/v1/battles/{battle_id}
```

前端调用：

```ts
api.battles.get(battleId)
```

## 6.4 录入双方阵容

```http
POST /api/v1/battles/{battle_id}/lineup
```

请求体：

```json
{
  "elves": [
    {
      "side": "self",
      "elf_id": "rocom_elf_0001",
      "build_id": "build_xxx",
      "is_active_elf": true
    },
    {
      "side": "enemy",
      "elf_id": "rocom_elf_0100",
      "build_id": null,
      "is_active_elf": true
    }
  ]
}
```

规则：

| 规则 | 当前行为 |
|---|---|
| 己方精灵 | 必须提供 `build_id` |
| 敌方精灵 | 只需要 `elf_id`，不需要资质、性格、技能配置 |
| 数量 | 每方最多 6 只，不强制必须 6 只 |
| 首发 | 每方最多 1 只首发 |
| 候选 | 敌方精灵会自动生成候选配置 |
| 阶段 | 正常只应在 `preparation` 阶段录入 |

响应：

```ts
interface LineupOut {
  battle_id: string;
  created_elf_state_count: number;
  generated_candidate_count: number;
  self_active_elf_id?: string | null;
  enemy_active_elf_id?: string | null;
}
```

前端调用：

```ts
api.battles.setupLineup(battleId, payload)
```

## 6.5 确认首发并开始战斗

```http
POST /api/v1/battles/{battle_id}/start
```

请求体：

```json
{
  "self_active_elf_id": "rocom_elf_0001",
  "enemy_active_elf_id": "rocom_elf_0100"
}
```

说明：如果阵容录入时已经设置首发，可以不传对应 ID。

前端调用：

```ts
api.battles.start(battleId, payload)
```

## 6.6 切换精灵

```http
POST /api/v1/battles/{battle_id}/switch
```

请求体：

```json
{
  "side": "self",
  "elf_id": "rocom_elf_0002",
  "turn_number": 3,
  "notes": "手动切换"
}
```

说明：切换会执行 `clear_on_switch` 状态清除规则，并生成相关事件/快照。

前端调用：

```ts
api.battles.switchElf(battleId, payload)
```

## 6.7 结束战斗

```http
POST /api/v1/battles/{battle_id}/finish
```

说明：设置为 `finished`，保留事件、候选、状态快照等历史数据。

前端调用：

```ts
api.battles.finish(battleId)
```

## 6.8 归档战斗

```http
POST /api/v1/battles/{battle_id}/archive
```

说明：设置为 `archived`，用于从最近战斗列表隐藏。不是物理删除。

前端调用：

```ts
api.battles.archive(battleId)
```

## 6.9 查询战斗完整状态

```http
GET /api/v1/battles/{battle_id}/state
```

响应：

```ts
interface BattleStateOut {
  battle: BattleOut;
  elves: Record<string, unknown>[];
  active_effects: Record<string, unknown>[];
  latest_snapshot_id?: string | null;
}
```

前端调用：

```ts
api.battles.state(battleId)
```

# 8. 伤害与资源事件接口

## 8.1 创建伤害事件

```http
POST /api/v1/battles/{battle_id}/damage-events
```

### 单次伤害请求

```json
{
  "turn_number": 1,
  "attacker_side": "self",
  "attacker_elf_id": "rocom_elf_0001",
  "defender_side": "enemy",
  "defender_elf_id": "rocom_elf_0100",
  "skill_id": "skill_xxx",
  "skill_confirmed": true,
  "damage_display_type": "single_damage",
  "damage_value": 120,
  "hp_percent_before": 100,
  "hp_percent_after": 70,
  "enemy_hp_percent_damage": 30,
  "notes": "手动录入"
}
```

## 15.2 当前仍属于占位或未实现的部分

| 项目 | 当前状态 | 建议 |
|---|---|---|
| 空库性格初始化 | 已实现启动时幂等自检查并写入/修正 30 种性格 | 启动前仍需先执行 Alembic 建表；该流程不插入示例精灵/技能/状态 |

| 事件重放重算 | 占位 | 后续实现 `EventReplayService` |
| 候选证据链 | 部分实现 | observation 已写入 evidence；聚合解释接口和前端详情展示仍需完善 |

## 15.3 建议下一步接口补充优先级

1. 状态定义导入接口/命令：支持 `effect_definition` JSON/CSV dry-run、commit、校验报告。
2. `GET /battles/{battle_id}/snapshots/{snapshot_id}` 前端封装与页面展示。
3. `POST /battles/{battle_id}/skill-events` 专用技能事件接口，减少前端手拼通用事件。
4. `POST /battles/{battle_id}/replay-from/{event_id}` 的真实重放实现。
5. 完善 `POST /observations/{battle_id}` 的状态触发、生存观测和更多规则分支 payload。
6. 完善候选 evidence 聚合接口和前端证据链展示。
7. 后续补充速度上下文接口，展示速度区间、先制/优先级和 unknown factors。

## 15.4 空库与状态定义数据约定

空库初始化当前拆成三层：

1. **结构层**：只由 Alembic 负责，执行 `python -m alembic upgrade head`。
2. **核心性格层**：后端启动时自动幂等自检查并写入/修正 30 种性格；该流程只处理 `nature_definition`。
3. **外部静态规则层**：精灵、技能、可学习技能、属性克制等 rocom 数据，通过 rocom 数据管线 dry-run 后再提交。

普通后端启动不会默认远程爬虫或大规模写库，也不会插入示例宠物/技能数据。历史 `app.seed.minimal_seed` 仅保留兼容用途，现在只补齐 30 种性格。

| 状态码 | 常见原因 |
|---:|---|
| 400 | 请求字段不满足业务规则，如己方阵容缺少 `build_id`、伤害类型缺少必填伤害字段、尝试物理删除未归档战斗。 |
| 403 | 管理接口令牌错误，`X-Admin-Token` 与 `ADMIN_UPDATE_TOKEN` 不匹配。 |
| 404 | 资源不存在，如 `battle_id`、`elf_id`、`skill_id`、`snapshot_id` 不存在，或已被物理删除。 |
| 409 | 数据更新任务已有运行中的任务，暂不接受新任务。 |
| 422 | JSON 格式或 Pydantic 字段类型校验失败。 |
| 502 | rocom 远程检查失败，通常是 BWIKI 网络或页面变化。 |
