> 创建日期：2026-05-19
> 修改日期：2026-05-27

> 当前定位：先固化数学口径、数据结构映射与代码落点；在未补齐验证样例前，不建议直接开启候选硬排除。

## 1. 设计目标

### 1.1 要解决的问题

本设计用于把附件中的“攻击伤害公式、状态伤害规则、印记触发规则”接入当前项目，形成可落地的后端计算与敌方配置反推方案。

需要明确的是：MVP 阶段玩家会手动录入屏幕上已经看到的伤害、扣血百分比、先后手和状态触发等事实。伤害计算模块的主要价值不是替代玩家读取屏幕，而是把这些观测事实转化为对敌方候选配置的约束、评分和证据链。

目标包括：

1. 将 PDF 中的公式整理为工程可执行的统一计算口径，用于解释玩家录入的观测事件。

2. 明确攻击伤害、状态伤害、印记伤害、特性伤害的分类边界。

3. 设计 `DamageFormulaContext` 的扩展结构，使计算器不再依赖散落字段。

4. 复用现有 `BattleEffectInstance` / `BattleEffectSnapshot` 状态系统，不另起天气、异常、印记专用系统。

5. 设计计算结果解释链，支撑前端展示和候选证据链。

6. 设计从“观测解释”到“候选软评分”再到“候选硬排除”的分阶段接入路径。

### 1.2 暂不解决的问题

以下内容不在 v0.1 中直接实现，先保留扩展位：

1. 所有精灵特性的完整特殊伤害处理。

2. 所有技能的特殊公式逐一落库。

3. 自动 OCR 识别伤害、状态、天气。

4. 未经验证的候选硬排除。

5. 前端复杂战斗回放模拟器。

## 2. 当前项目中的接入点

当前代码中已经有较好的预留点：

|模块|当前职责|本次集成后的职责|
|---|---|---|
|`backend/app/calculation/damage_calculator.py`|伤害公式占位器|真实伤害计算总入口|

|`backend/app/models/static.py::EffectDefinition`|统一状态定义|存储公式钩子、属性修正、增伤/减伤、状态结算规则|

## 3. 术语与伤害来源分类

### 3.1 伤害来源

一个回合内，伤害来源分为三类：

|来源|说明|是否有统一公式|建议处理器|
|---|---|---|---|
|攻击伤害|技能直接造成的物攻/魔攻伤害，可能有连击|有|`AttackDamageCalculator`|
|状态伤害|灼烧、中毒、寄生、冻结、棘刺等状态或印记结算|有|`StatusDamageCalculator`|
|特性伤害|精灵特性直接造成的伤害|无|`TraitDamageHandlerRegistry`|

注意：星陨印记虽然来自印记，但 PDF 根据伤害逻辑与特性判断，符合定义，为“攻击伤害”，应走攻击公式的简化分支，而不是普通状态百分比伤害。

### 3.2 战斗阶段

公式和状态结算需要区分：

1. **回合前期**：切换、上场触发、特性开始生效、棘刺触发。

2. **回合中期**：根据先手与速度释放技能，攻击伤害和部分即时状态结算发生在这里。

3. **回合末期**：灼烧、中毒、寄生等持续状态结算；状态结算，状态层数变化；天气变化结算。

状态结算顺序原则：**同一阶段内，按状态获得顺序结算。** 当前 `BattleEffectInstance.applied_turn` 不足以表达同回合内顺序，后续如要严格结算，建议增加或在 payload 中保存 `applied_order` / `source_event_order`。

## 4. 数学公式标准化

### 4.1 数值约定

工程实现中统一使用以下约定：

|概念|工程表示|
|---|---|
|100%|`1.0`|
|50%|`0.5`|
|2% 最大生命|`max_hp * 0.02`|
|本系加成 25%|`1.25`|
|减伤 90%|`damage_reduction = 0.90`，最终乘 `1 - 0.90`|
|克制|`2.0`|
|抵抗|`0.5`|
|双重克制|`3.0`|
|双重抵抗|`1 / 3`|

建议计算内部使用 `Decimal`，避免浮点误差；最终输出再转 `int` / `float`。

### 4.2 能力等级

PDF 公式：

```text
能力等级 =
(1 + 进攻方攻击提升 + 防御方防御降低)
/
(1 + 进攻方攻击降低 + 防御方防御提升)
```

工程字段建议：

```text
stat_stage_multiplier =
(1 + attacker_attack_up + defender_defense_down)
/
(1 + attacker_attack_down + defender_defense_up)
```

其中：

- 物理攻击技能读取物攻提升/降低、物防提升/降低。

- 魔法攻击技能读取魔攻提升/降低、魔防提升/降低。

- 状态来源应为 `BattleEffectSnapshot` 中的属性修正类状态。

- 一层“力量增效”如对应 100%，则写作 `1.0`。

### 4.3 显示威力

```text
显示威力 =
(技能威力 × 应对倍率 + 威力加成)
× 威力提升状态增益
× 能力等级
× 本系加成
× 克制关系
× 天气影响
```

工程建议拆为：

```text
display_power =
(base_power * response_multiplier + flat_power_bonus)
* power_boost_multiplier
* stat_stage_multiplier
* stab_multiplier
* type_effectiveness_multiplier
* weather_multiplier
```

字段来源：

|项|来源|
|---|---|
|`base_power`|`SkillDefinition.base_power` 或 `damage_rule_json.base_power_override`|
|`response_multiplier`|技能规则，应对成功时由输入或状态判定|
|`flat_power_bonus`|战场信息/状态技能使用日志/技能槽状态|
|`power_boost_multiplier`|进攻方状态栏中的导致威力提升的状态增益|
|`stat_stage_multiplier`|见 4.2|
|`stab_multiplier`|技能系别与进攻方精灵系别；PVP 先按 1.25|
|`type_effectiveness_multiplier`|技能系别 vs 防御方系别|
|`weather_multiplier`|天气状态与技能天气规则|

说明：PDF 强调“不稳定加成不包含在显示威力内”，因此不稳定加成必须在最终攻击伤害中单独处理。

### 4.4 属性克制倍率

单属性防御：读取 `type_effectiveness_rule`。

双属性防御不能简单把两个倍率相乘，因为 PDF 给出的双重克制/抵抗规则为：

|情况|倍率|
|---|---|
|两个防御属性均被克制|`3.0`|
|两个防御属性均抵抗|`1 / 3`|
|一个克制、一个抵抗|`1.0`|
|只对其中一个存在克制|`2.0`|
|只对其中一个存在抵抗|`0.5`|
|都无关系|`1.0`|

建议实现专门函数：

```python
def resolve_type_multiplier(attack_type: str, defense_types: list[str]) -> Decimal:
    ...
```

不要直接做 `multiplier_a * multiplier_b`。

### 4.5 本系加成

PVP 当前按固定：

```text
stab_multiplier = 1.25
```

触发条件：技能系别属于进攻方精灵系别之一。

后续如需要成长星级差异，可改为从精灵配置或规则表读取。

### 4.6 连击处理

PDF 中给出的连击公式存在一定歧义，当前工程建议分两层处理：

#### 4.6.1 伤害显示类型层

沿用现有 `DamageDisplayType`：

|类型|计算/记录方式|
|---|---|
|`single_damage`|单次伤害，理论伤害与 `damage_value` 比较|
|`visual_total_damage`|动画多段但显示总伤害，按一次总伤害比较|
|`combo_repeated_damage`|每段相同，无总伤害显示，优先用单段伤害比较|
|`special_damage`|交给特殊处理器|

#### 4.6.2 连击数层

把“是否连击”和“连击数”拆开：

```json
{
  "is_combo": true,
  "hit_count_type": "fixed | effect_modified | skill_specific | manual",
  "base_hit_count": 2,
  "combo_bonus_source": "effect | trait | manual",
  "mud_roar_fang_blocked": false,
  "per_hit_same_damage": true
}
```

候选过滤初期：

- `combo_repeated_damage` 优先比较单段理论伤害与 `per_hit_damage_value`。

- 总伤害只用于生命扣减和击杀判断。

- 连击次数未知或由特性影响时，不硬排除。

### 4.7 攻击伤害公式

PDF 公式整理为：

```text
攻击伤害计算 =
(进攻方攻击属性值 / 防御方防御属性值 × 37 / 41)
× (显示威力 × 不稳定加成)
× 连击
× ∏(1 - 减伤_i)

攻击技能伤害 = int(攻击伤害计算)
```

工程表示：

```text
raw_attack_damage =
(attacker_offense_stat / defender_defense_stat)
* (37 / 41)
* display_power
* unstable_multiplier
* hit_multiplier_or_count
* damage_reduction_product

final_damage = floor(raw_attack_damage)
```

其中：

- `attacker_offense_stat`：物攻或魔攻。

- `defender_defense_stat`：物防或魔防。

- `unstable_multiplier`：未知时默认为范围 `[1.0, unknown_max]` 或标记 `unknown`，不建议硬排除候选。

- `damage_reduction_product = ∏(1 - reduction_i)`。

- `int()` 在 Python 对正数等价于向下取整。

## 5. 状态与印记结算规则

### 5.1 状态规则维度

每个状态/印记至少从四个维度描述：

1. **变化**：层数如何变化，是否随回合变化。

2. **结算**：在哪个阶段结算，是否造成伤害或回复。

3. **特性/免疫**：是否受属性免疫、是否有特殊增益免疫。

4. **去除**：切换、净化、特殊技能是否去除。

这些规则应落到 `EffectDefinition` 的 JSON 字段中，而 `BattleEffectInstance` 保存当前层数、归属、来源、是否生效。

### 5.2 灼烧

|项|规则|
|---|---|
|分类|异常状态，状态伤害|
|结算阶段|回合末期；部分技能可在回合中期额外触发即时结算|
|伤害公式|`max_hp × 0.02 × layers × fire_type_multiplier`|
|伤害属性|火系伤害，计算克制|
|减伤|暂不参与减伤|
|免疫|火系精灵免疫灼烧状态|
|层数变化|伤害结算后层数减半，向下取整|
|去除|常规异常状态去除；切换清除|

即时结算特例：如“充分燃烧”导致回合中期结算一次伤害，此次文档说明“不属于火系伤害，无法被抵抗”，应通过 `special_rule_id` 或技能 `effect_operations_json` 标记。

### 5.3 中毒

|项|规则|
|---|---|
|分类|异常状态 + 中毒印记|
|结算阶段|回合末期|
|结算顺序|先中毒印记，后中毒状态|
|伤害公式|`max_hp × 0.03 × layers × poison_type_multiplier`|
|伤害属性|毒系伤害，计算克制|
|减伤|暂不参与减伤|
|免疫|毒系精灵免疫中毒状态，但仍受中毒印记伤害|
|状态层数变化|默认不变|
|印记变化|适用印记规则|
|去除|中毒状态可常规净化；中毒印记只能由印记去除规则处理|

裘卡等“状态转印记”的转换时点在回合末期且中毒状态伤害后，建议作为 `special_rule_id` 处理。

### 5.4 寄生

|项|规则|
|---|---|
|分类|异常状态，状态伤害|
|结算阶段|回合末期|
|伤害公式|`max_hp × 0.06`|
|伤害属性|真实伤害|
|克制|不计算克制|
|减伤|暂不参与减伤|
|免疫|草系精灵免疫寄生状态|
|回复|将造成的伤害作为恢复效果给对方回血|
|层数变化|默认不变|
|去除|常规异常状态去除；切换清除|

建议计算结果返回 `secondary_events`：

```json
{
  "damage": 18,
  "secondary_events": [
    {"type": "heal", "target": "source_or_opponent", "value": 18}
  ]
}
```

### 5.5 冻结

|项|规则|
|---|---|
|分类|异常状态，触发型真实伤害|
|触发时机|任意阶段；当生命百分比小于等于冻结阈值时触发|
|冻结阈值|每层 5% 生命比例，即 `layers * 5%`|
|效果|触发后直接导致力竭|
|伤害属性|真实伤害 / 阈值斩杀型结算|
|克制/减伤|不计算克制，不参与减伤|
|去除|被去除后不造成伤害|
|免疫|冰系精灵免疫获得冻结状态，获得特性免疫冻结但不能移除已存在冻结状态|

工程上不要把冻结简单实现为普通扣血，而应实现为阈值检查：

```text
freeze_threshold_percent = layers * 5%
if hp_percent_after_damage <= freeze_threshold_percent:
    defeated_by_freeze = true
```

与数学建模保持一致，v0.1 固定使用 `<=` 作为触发条件。

### 5.6 棘刺

|项|规则|
|---|---|
|分类|印记，状态伤害|
|触发阶段|回合前期，新精灵上场时|
|伤害公式|`max_hp × 0.06 × layers`|
|伤害属性|真实伤害|
|克制/减伤|不计算克制，不参与减伤|
|层数变化|适用印记规则|
|去除|适用印记去除规则|

切换流程应为：

1. 离场精灵清除可切换清除的异常状态。

2. 新精灵上场。

3. 棘刺印记在回合前期触发。

4. 创建 `DamageEvent` 或 `ResourceChangeEvent`，并绑定快照。

### 5.7 星陨

|项|规则|
|---|---|
|分类|印记，但伤害类型属于攻击伤害|
|触发条件|非幻属性攻击技能命中或作用于防御方时触发；状态技能/防御技能默认不触发，除非后续规则库另行声明|
|触发时机|攻击技能生效后；若防御方未力竭且满足触发条件，再承受星陨伤害|
|星陨威力|`layers^2 + layers × 24 - 24`|
|攻防属性选择|跟随触发技能类别：物理技能用物攻/物防，魔法技能用魔攻/魔防|
|结算伤害属性|星陨结算伤害属于幻系；但触发条件要求本次触发攻击技能本身不是幻系|
|伤害公式|攻击公式简化分支，不包含显示威力、本系、克制、天气、不稳定加成|
|减伤|会被防御技能减伤|
|去除|触发后按印记规则去除；暮星辰特例为层数减半向上取整|

星陨伤害公式整理为：

```text
starfall_power = layers^2 + layers * 24 - 24

starfall_damage = int(
  (trigger_offense_stat / trigger_defense_stat * 37 / 41)
  * starfall_power
  * product(1 - reduction_i)
)
```

说明：

1. `trigger_offense_stat / trigger_defense_stat` 根据触发技能类别选择物攻/物防或魔攻/魔防。

2. 星陨结算伤害记录为幻系伤害，但公式不乘属性克制、本系、天气或显示威力。

3. PDF 根据系统逻辑判断简化公式不存在不稳定加成，v0.1 不继承攻击方不稳定加成。

4. 星陨攻防属性只借用触发技能类别，不继承触发技能的属性克制链。

### 6.6 条件应对规则 `conditional_effects`

“应对成功”不应只建模为一个伤害倍率，因为大量攻击技能、状态技能、防御技能会在应对成功后产生不同影响，例如增伤、减伤、施加状态、回复、反击、改变连击、取消对方技能等。

因此建议在规则数据中增加通用条件分支结构。短期仍可放在 `SkillDefinition.damage_rule_json` 或 `effect_operations_json` 内，暂不强制新建表：

```json
{
  "conditional_effects": [
    {
      "branch_id": "response_success",
      "trigger": "response_success",
      "conditions": {
        "target_action_type": "attack",
        "manual_required_when_unknown": true
      },
      "operations": [
        {
          "type": "modify_response_multiplier",
          "value": 3.0
        },
        {
          "type": "apply_effect",
          "effect_id": "effect_xxx",
          "target": "defender",
          "layers": 1
        }
      ]
    }
  ]
}
```

规则解析后应输出统一操作，而不是让伤害公式直接理解“应对成功”：

```text
RuleResolver output:
  response_multiplier
  flat_power_bonus
  power_boost_multiplier
  damage_reductions
  hit_count_modifier
  effect_operations
  unknown_factors
```

长期如果规则数量增多、需要前端维护或按条件查询，再考虑拆出独立表，例如 `skill_conditional_rule`：

```text
rule_id
skill_id
trigger_type
condition_json
operation_json
priority
data_version
developer_notes
```

v0.1 建议先维护 JSON 规则库，配合 dry-run 校验，避免过早建表导致频繁迁移。

## 7. `DamageFormulaContext` 扩展设计

当前 `DamageFormulaContext` 只记录事件 ID、双方、技能、快照等粗字段。真实计算建议扩展为 v2：

```python
class DamageFormulaContext(BaseModel):
    battle_id: str
    damage_event_id: str | None = None
    battle_event_id: str | None = None
    snapshot_id: str | None = None

    formula_type: str  # attack | status | starfall | trait | special

    attacker_side: str | None = None
    attacker_elf_id: str | None = None
    attacker_element_types: list[str] = []
    attacker_panel_stats: dict | None = None

    defender_side: str | None = None
    defender_elf_id: str | None = None
    defender_element_types: list[str] = []
    defender_panel_stats: dict | None = None
    defender_current_hp: int | None = None
    defender_max_hp: int | None = None
    defender_hp_percent: float | None = None

    skill_id: str | None = None
    skill_element_type: str | None = None
    skill_category: str | None = None
    base_power: int | None = None
    damage_rule: dict | None = None
    hit_rule: dict | None = None

    active_effects: list[dict] = []
```

说明：

- v2 不一定要一次性落库成表字段，可以继续存入 `formula_context_json`。

- `attacker_panel_stats` 和 `defender_panel_stats` 对己方可以来自 `BattleElfState`，对敌方候选过滤时来自 `BuildCandidate`。

- `manual_flags` 用于保存本次用户手动判定：应对是否成功、连击次数、天气是否生效、不稳定加成是否已知等。

## 9. 代码模块设计

建议拆分：

```text
backend/app/calculation/
  damage_calculator.py          # 总入口：根据 formula_type 分发
  formula_context.py            # 上下文与结果模型
  attack_damage.py              # 攻击伤害公式
  rule_resolver.py             # 应对/条件分支规则解析，输出公式修正项和状态操作
  display_power.py              # 显示威力公式
  status_damage.py              # 灼烧/中毒/寄生/冻结/棘刺
  starfall_damage.py            # 星陨印记攻击分支
  type_effectiveness.py         # 克制倍率解析
  modifier_resolver.py          # 状态、天气、能力等级、减伤、增伤解析
  rounding.py                   # 取整策略
  calculation_trace.py          # 解释链工具，可选
```

### 9.1 总入口

```python
class DamageCalculator:
    def calculate(self, context: DamageFormulaContext) -> DamageCalculationResult:
        if context.formula_type == "attack":
            return AttackDamageCalculator(...).calculate(context)
        if context.formula_type == "status":
            return StatusDamageCalculator(...).calculate(context)
        if context.formula_type == "starfall":
            return StarfallDamageCalculator(...).calculate(context)
        if context.formula_type == "trait":
            return TraitDamageHandlerRegistry(...).calculate(context)
        return DamageCalculationResult(status="formula_unavailable", ...)
```

### 9.2 依赖解析服务

伤害计算本身尽量做纯函数，不直接查数据库。建议由服务层构造完整上下文：

```text
DamageEventService / InferenceEngine
  -> DamageFormulaContextBuilder
      -> 读取 SkillDefinition
      -> 读取 ElfDefinition
      -> 读取 BattleElfState 或 BuildCandidate
      -> 读取 BattleEffectSnapshot
      -> 组装 DamageFormulaContext
  -> DamageCalculator.calculate(context)
```

这样更容易单元测试，也方便对候选批量计算。

## 10. 候选推算接入策略

### 10.1 反推主线

伤害计算与状态结算接入候选推算时，应以“敌方配置反推”为主线：

```text
玩家录入观测事件
  -> 构造 DamageFormulaContext / ObservationContext
  -> 对每个 BuildCandidate 替换敌方面板属性
  -> 计算该候选下的理论伤害、速度关系或状态触发结果
  -> 与玩家观测比较
  -> 更新 match_score / confidence / evidence
```

因此，`DamageCalculator`、`StatusDamageCalculator`、`StarfallDamageCalculator` 都是候选反推系统的观测解释器。

### 10.2 初期只做软评分

在公式样例不足前，不建议直接 `is_excluded = true`。推荐策略：

|结果|行为|
|---|---|
|理论结果命中观测值|`match_score += weight`，记录 matched event|
|理论结果不命中但存在未知因素|不排除，记录 warning / unknown factor|
|理论结果明显不命中且无未知因素|`match_score -= weight`，先不排除|
|多次高可信证据均不命中|后续再考虑硬排除|

### 10.3 观测类型与比较策略

MVP 中建议至少支持以下观测：

|观测类型|比较对象|说明|
|---|---|---|
|整数伤害|`observed_damage_value` vs 理论伤害|玩家直接读到伤害数值时最可靠|
|扣血百分比|`hp_percent_delta` vs `damage / max_hp * 100`|需要 UI 读数容差|
|先后手|我方速度 vs 候选速度|存在优先级或速度状态时降级为软约束|
|技能出现|技能是否在候选技能池|技能池数据完整时可作为强证据|
|状态触发|冻结、星陨、灼烧、中毒等是否可解释|依赖状态快照和层数|
|存活/力竭|伤害后 HP 是否大于 0|当前 HP 不确定时只做弱约束|

百分比比较需要容忍误差：

```text
percent_tolerance = 0.5% ~ 1.0%
```

原因：游戏 UI 百分比可能四舍五入或只显示整数。

### 10.4 何时允许硬排除

满足以下条件才建议硬排除：

1. 观测事实可靠，例如整数伤害、明确技能出现或明确状态触发。

2. 技能已确认。

3. 攻防双方精灵已确认。

4. 伤害显示类型明确。

5. 参与公式的状态快照完整。

6. 不稳定加成不存在或已确认。

7. 防御方 HP 值或观测伤害值可靠。

8. 连击次数可靠。

9. 公式已通过样例测试。

否则只做软评分。

## 11. 取整规则

v0.1 暂定：

|场景|取整|
|---|---|
|攻击伤害最终值|`floor`，即 Python 正数 `int()`|
|显示威力|不提前取整，除非后续样例证明游戏中间取整|
|状态百分比伤害|先计算 raw，再最终 `floor`|
|灼烧层数减半|`floor(layers / 2)`|
|暮星辰星陨层数减半|`ceil(layers / 2)`|
|面板属性|沿用当前 `StatCalculator`|

待验证项：

1. 游戏是否对显示威力进行中间取整。

2. 百分比状态伤害是否最小为 1。

3. 多段连击是否每段分别取整后求和。

## 15. 待确认问题

以下口径已按数学建模文档明确，后续不再作为待确认项：

- 冻结触发条件固定为 `hp_percent <= layers * 5%`。

- 星陨攻防属性选择跟随触发技能类别：物理技能用物攻/物防，魔法技能用魔攻/魔防。

- 星陨结算伤害记录为幻系，但公式不额外乘显示威力、本系、属性克制、天气或不稳定加成。

剩余问题会影响最终计算准确性，建议在实现前或实现早期用样例验证：

1. 攻击公式中的 `int()` 是否只在最终取整，还是中间也取整。

2. 显示威力是否一定等于游戏 UI 中显示的值，是否已经取整。

3. 状态百分比伤害是否按最大生命计算，是否存在最小 1 伤害。

4. 灼烧、中毒等百分比伤害是否最终向下取整。

5. 星陨触发后普通情况是清空印记还是按特殊规则变化。

6. 连击公式中“是否不存在泥吼牙”的具体判定是否需要纳入通用规则。

7. 不稳定加成未知时的候选比较范围应如何设置。

8. 天气影响是否按技能系别统一倍率，还是每个技能独立配置。

## 16. v0.1 推荐结论

1. 当前项目已有的状态快照与伤害事件设计可以直接承接本次公式，不需要重做架构。

2. 第一轮实现应先做纯计算、观测解释与证据链，不要直接污染候选排除。

3. 状态伤害和攻击伤害应分离；星陨虽然是印记，但走攻击伤害分支。

4. 所有不确定因素应进入 `unknown_factors`，只要存在未知因素，就不做候选硬排除。

5. 在积累样例前，候选反推先采用软评分，核心目标是稳定输出候选分布与证据链。
